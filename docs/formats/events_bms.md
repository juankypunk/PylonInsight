# SC0500A BMS Event CSV Format

## Purpose

This document describes the CSV format generated by BatteryView 3.9.4 when exporting the event log from a Pylontech SC0500A Battery Management System (BMS).

---

## Supported Device

The format described in this document has been validated using the following hardware:

```
Device name         : CMU_A
Board version       : TISP01V10R02_1
Main Soft version   : B52.36.0
Soft version        : V5.7
```

BatteryView version:

```
3.9.4
```

## File Structure

The file consists of:

* Header
* CSV column names
* Event records
* Footer

Unlike `history.csv`, this file stores event-driven snapshots rather than periodic samples.

---

## Header

The first lines identify the export type.

Example:

```text
datalist,event,0
```

### Header inconsistency

BatteryView omits the `Date` column from the CSV header.


---

## Record Structure

Each record contains the complete operating state of the BMS at the instant an event was recorded.

Unlike the history export, records are generated only when BatteryView decides that an event is worth logging.

---

## Timestamp

The timestamp is split into two CSV columns:

* Date
* Time

Applications should reconstruct a single timestamp by combining both fields.

---

## Event Model

Each record contains:

* instantaneous electrical measurements
* operating state
* status flags
* event flags

This allows the operating conditions that triggered the event to be reconstructed.

---

## Primary Key

The recommended natural key is:

```
(Date, Time)
```

The `Item` field is only a sequential index inside the exported file and must not be used as a persistent identifier.

---

## Relationship with History

`history.csv`

* periodic sampling
* fixed interval

`event.csv`

* event-driven sampling
* irregular interval

Both formats share a very similar record structure.

---

## Known Event Codes

Observed event codes include:

* BHV
* BLV
* DSG

Their exact meaning is currently under investigation.

## Related findings

## F-014 — event.csv header omits the Date column

**Status:** ✅ Confirmed

**Affected exports**

✅ SC0500A BMS
✅ BMU
✅ XHB_BMU_NT

**Description**

All event.csv files exported by BatteryView omit the Date column in the CSV header.

However, every record contains both a date and a time field.

Therefore, the effective header is:

Item,
Date,
Time,
...

instead of:

Item,
Time,
...

**Evidence**

Header:

Item,Time,Vo(mV),...

Record:

0,26-03-06,20:18:31,53936,...

**Parser implications**

Importers should detect that:

number_of_fields(record) = number_of_columns(header) + 1

and insert the missing Date column before parsing.
